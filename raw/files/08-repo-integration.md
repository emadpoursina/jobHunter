# 08 — Repo Integration

The backend reads from and writes to the existing `jobHunter` repo file
structure. This document defines exactly which files are touched and how.

---

## Principle

The backend is a **guest** in the existing repo. It:
- Reads: `profile/master-profile.md`, `agents/*.md`
- Writes: `job-offers/by-country/<CC>/<slug>.md`, `documents/generated/CV_*.md`
- Never modifies or deletes existing files

---

## Files the backend reads

### `profile/master-profile.md`

Read by `pipeline/cv.js` on every CV generation call.

- Full contents are injected into the LLM user message as the candidate profile
- If the file is missing, throw an `LLM_ERROR`: "Master profile not found at
  profile/master-profile.md. Please create this file."

### `agents/cv-generator.md`

Read by `pipeline/cv.js` as the LLM system prompt for CV generation.

- If missing, fall back to the default prompt defined in `05-pipeline.md`
- Log a warning when the fallback is used

### `agents/job-offer-research.md`

Read by `pipeline/parser.js` as the base system prompt for offer parsing.

- If missing, fall back to the default parser prompt defined in `05-pipeline.md`
- Log a warning when the fallback is used

---

## Files the backend writes

### Offer Markdown files

**Path pattern:**
```
job-offers/by-country/<CC>/<slug>.md
```

Where:
- `<CC>` is the ISO 3166-1 alpha-2 country code (uppercase), e.g. `DE`, `NL`
- `<slug>` is generated by `base.makeSlug(company, title, date)`, e.g.
  `acme-gmbh-backend-engineer-2025-06`

**Directory creation:** Create the `<CC>/` subdirectory if it doesn't exist
(`fs.mkdir(..., { recursive: true })`).

**File contents** — follow the existing `_offer-template.md` format from the
repo. The parser writes this after a successful LLM parse:

```markdown
---
date_collected: <YYYY-MM-DD>
source: <collector name>
source_url: <url or "manual">
status: new
match_score: <0-100>
---

# <title> — <company>

## Meta
| Field | Value |
|-------|-------|
| Company | <company> |
| Role | <title> |
| Location | <location> |
| Type | <employment_type> |
| Salary | <salary or "Not listed"> |
| Visa sponsorship | <visa_sponsorship> |
| Match score | <match_score>% |

## Required skills
<required_skills as bullet list>

## Nice to have
<nice_to_have as bullet list>

## Key responsibilities
<responsibilities as bullet list>
```

**Collision handling:** If the file already exists (same slug), append a
numeric suffix: `acme-gmbh-backend-engineer-2025-06-2.md`. This should be rare
since deduplication happens first.

---

### CV Markdown files

**Path pattern:**
```
documents/generated/CV_<CompanySlug>_<RoleSlug>_<YYYY-MM-DD>.md
```

Where slugs are title-cased, spaces replaced with empty string:
```
CV_AcmeGmbH_BackendEngineer_2025-06-01.md
```

**Directory:** `documents/generated/` already exists in the repo.

**File contents:** Exactly the Markdown string returned by the LLM — no
additional header or frontmatter added. The LLM output is the complete file.

**Collision handling:** If the file already exists, overwrite it (the user may
regenerate a CV for the same job).

---

## Path resolution

All repo file paths are resolved relative to `REPO_ROOT` from the environment.

```js
import { resolve } from 'path'
const REPO_ROOT = resolve(process.env.REPO_ROOT ?? '.')

export function repoPath(...parts) {
  return resolve(REPO_ROOT, ...parts)
}
```

When the server entry point is `server/index.js` and `REPO_ROOT=..`, all paths
resolve correctly relative to the repo root.

---

## Existing file conventions to preserve

The repo uses these conventions — the backend must not deviate from them:

| Convention | Rule |
|-----------|------|
| Offer filenames | `<slug>.md` — all lowercase, hyphens |
| CV filenames | `CV_<Company>_<Role>_<Date>.md` — mixed case, underscores |
| Frontmatter | YAML-style `---` delimited header at top of offer files |
| Date format | `YYYY-MM-DD` everywhere |
| Country folders | Uppercase ISO code, e.g. `DE/`, `NL/` |

---

## `data/` is git-ignored

Add to `.gitignore`:
```
data/
.env
```

The SQLite database is runtime state, not source-controlled. The repo files
(offers, CVs) are source-controlled because they are the human-readable outputs.
