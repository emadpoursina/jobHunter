# 05 — Pipeline

The pipeline has two stages: **Parser** and **CV Generator**. Both are pure
async functions that take inputs, call the LLM via `server/llm.js`, and return
structured outputs. They do not write to the DB or file system — the calling
route handler does that.

---

## Parser (`pipeline/parser.js`)

### Purpose
Converts a `RawOffer` (unstructured text) into a `StandardOffer` (structured
JSON matching the project's offer schema).

### Function signature
```js
export async function parseOffer(rawOffer: RawOffer): Promise<StandardOffer>
```

### StandardOffer schema
```js
{
  title:            string,
  company:          string,
  location:         string,         // "City, Country"
  countryCode:      string,         // ISO 3166-1 alpha-2, derived from location
  employmentType:   string,         // "Full-time" | "Part-time" | "Contract" | "Remote"
  salary:           string | null,
  visaSponsorship:  string,         // "Yes" | "No" | "Not mentioned"
  requiredSkills:   string[],
  niceToHave:       string[],
  responsibilities: string[],
  matchScore:       number          // 0–100, against the candidate profile
}
```

### LLM system prompt

The parser reads the file `agents/job-offer-research.md` from the repo and
uses its contents as the base system prompt. It then appends the following
instruction block:

```
---
STRUCTURED EXTRACTION TASK

You will receive the raw text of a job listing. Extract the information and
return ONLY a valid JSON object with exactly these keys — no markdown fences,
no preamble, no explanation:

{
  "title": "Job title",
  "company": "Company name",
  "location": "City, Country",
  "countryCode": "DE",
  "employmentType": "Full-time | Part-time | Contract | Remote",
  "salary": "Range as string, or null",
  "visaSponsorship": "Yes | No | Not mentioned",
  "requiredSkills": ["skill1", "skill2"],
  "niceToHave": ["skill1"],
  "responsibilities": ["resp1", "resp2"],
  "matchScore": <integer 0-100>
}

For matchScore, evaluate how well this candidate profile matches the role:
<CANDIDATE_SUMMARY>
```

The `<CANDIDATE_SUMMARY>` placeholder is replaced at runtime with the first
400 characters of `profile/master-profile.md`.

### LLM user message
The raw text from `rawOffer.rawText`.

### JSON parsing
The LLM response may contain markdown fences despite the instruction. Strip
them before parsing:
```js
const clean = responseText.replace(/```json|```/g, '').trim()
const offer = JSON.parse(clean)
```

If `JSON.parse` throws, throw an error with code `PARSE_ERROR` that includes
the first 200 chars of the raw LLM response for debugging.

### Post-processing
After parsing:
1. Validate all required fields are present — fill missing strings with `"Unknown"`
   and missing arrays with `[]`
2. If `countryCode` is missing or invalid, derive it using `base.inferCountryCode(offer.location)`
3. Clamp `matchScore` to the range 0–100

## CV Generator (`pipeline/cv.js`)

### Purpose
Takes a `StandardOffer` + the candidate's master profile and generates a
tailored CV in Markdown format.

### Function signature
```js
export async function generateCv(job: DbJob): Promise<string>
```
`DbJob` is the full database record (already has all parsed fields).

### LLM system prompt

Reads `agents/cv-generator.md` from the repo and uses its full contents as the
system prompt. This file is the human-authored instruction for the CV agent —
it describes tone, structure, formatting rules, and tailoring strategy. Cursor
should not hard-code CV instructions; they live in the repo agent file.

If `agents/cv-generator.md` does not exist, fall back to this minimal prompt:
```
You are an expert CV writer for software engineers. Generate a tailored,
professional CV in Markdown format. The CV must:
- Start with a focused summary that directly addresses the role
- List only the most relevant skills
- Reframe experience and project bullet points to match the job
- Keep it to 1–1.5 pages (400–600 words)
- Output Markdown only — no preamble, no code fences
```

### LLM user message

```
CANDIDATE PROFILE:
<contents of profile/master-profile.md>

---

TARGET JOB:
Title: <job.title>
Company: <job.company>
Location: <job.location>
Type: <job.employment_type>
Visa sponsorship: <job.visa_sponsorship>
Required skills: <job.required_skills joined with ", ">
Nice to have: <job.nice_to_have joined with ", ">
Key responsibilities: <job.responsibilities joined with "; ">

Generate the tailored CV now.
```

### Output
Returns the raw Markdown string from the LLM response. The calling route
handler writes it to disk and updates the DB record.

### File naming
The CV file name is generated by the route handler (not this module), following
the convention from the existing repo:
```
CV_<CompanySlug>_<RoleSlug>_<YYYY-MM-DD>.md
```
Written to `documents/generated/`.

## Reading repo files

Both pipeline modules need to read files from the repo. Use this helper
(implement in `pipeline/repoFiles.js` or inline in each module):

```js
import { readFile } from 'fs/promises'
import { resolve }  from 'path'

const REPO_ROOT = resolve(process.env.REPO_ROOT || '.')

export async function readRepoFile(relativePath) {
  const fullPath = resolve(REPO_ROOT, relativePath)
  try {
    return await readFile(fullPath, 'utf8')
  } catch {
    return null   // caller decides what to do with missing files
  }
}
```

`REPO_ROOT` is the path to the jobHunter repo root. When the server runs from
`server/`, it is `'..'`. When it runs from the repo root, it is `'.'`.
Set it in `.env` as `REPO_ROOT=.` (default).

## Token budget

Ollama local models have a default context limit of ~4096 tokens for smaller
models. Keep inputs within budget:

| Input | Approximate tokens |
|-------|--------------------|
| Raw offer text | 500–1500 |
| Master profile | 400–800 |
| Agent instruction file | 200–500 |
| Total for parser | ~2500 max |
| Total for CV generator | ~2500 max |

If `rawText` exceeds 3000 characters, truncate to 3000 before sending to the
parser. Log a warning when this happens.

Set `max_tokens: 2000` for CV generation (CV output is longer than parse output).
Set `max_tokens: 800` for parsing.
